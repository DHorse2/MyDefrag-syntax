# Procedure - Markdown Document Update

## Purpose

Update, revise, or create Markdown documentation while preserving technical accuracy, document intent, applicable standards, and requested structure.

## Applicability

Use this procedure when the task involves creating, editing, correcting, restructuring, or validating a Markdown document.

This procedure is especially applicable to:

- Project documentation.
- Standards documents.
- Role documents.
- Procedure documents.
- Architecture documents.
- Status reports.
- Execution summaries.

## Prerequisites

Before using this procedure:

- The current task must require Markdown documentation work.
- The active role must have authority to create or revise documentation.
- The relevant source document or requested output scope must be known.
- Applicable documentation standards must be identified.

## Inputs

Typical inputs include:

- User instructions.
- Existing Markdown document.
- Requested changes.
- Applicable standards.
- Applicable procedures.
- Relevant project context.
- Prior execution records, when available.

## Required Standards

Load only the standards required for the current task, for example:

- Global Markdown Documentation Standard.
- Output Document Formatting Rules.
- Execution Record Standard, when an execution record is required.

## Invoked Procedures

This procedure may invoke:

- `Validation.md`
- `Execution_Record.md`

Load invoked procedures only when required by the task and only once per run.

Do not recursively load procedures beyond one level unless explicitly instructed.

## Steps

1. Identify the target document or requested new document.
2. Confirm the requested change, output scope, and applicable standards.
3. Inspect the existing document when revising an existing file.
4. Preserve existing structure, level of detail, and intent unless the user explicitly requests a structural change.
5. Apply the requested documentation change.
6. Use one level-1 heading for the document.
7. Ensure headings are unique.
8. Use `-` for unordered list items.
9. Prefer tables where they improve clarity.
10. Avoid inventing technical facts.
11. Identify all created, modified, or affected files.
12. Run or describe validation appropriate to the task.
13. Produce an execution record when required.

## Expected Outputs

Typical outputs include:

- Updated Markdown document.
- New Markdown document.
- Revision summary.
- List of created, modified, or affected files.
- Validation summary.
- Execution record, when required.

## Validation

Validation should confirm that:

- The requested documentation change is complete.
- The document has exactly one level-1 heading.
- Headings are unique.
- Unordered lists use `-` rather than `*`.
- Required sections are present.
- Existing structure was preserved unless explicitly changed.
- Created, modified, or affected files are identified.
- Applicable standards were followed.

## Completion Criteria

This procedure is complete when:

- The Markdown document has been created or updated as requested.
- Required validation has been completed or explicitly reported as unavailable.
- The response identifies the output file or document.
- Any required execution record has been created or described.

## Notes

This procedure defines **how** Markdown documentation work is performed. It does not define **who** performs the work. The active role defines authority and responsibility, while standards define the rules this procedure must satisfy.
